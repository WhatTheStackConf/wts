# MCP progressive authorization research

Date: 2026-07-13

## Scope

This note evaluates protocol and SDK support, not which behavior WTS should choose. It uses the MCP `2025-11-25` specification, currently marked latest, and TypeScript SDK `v1.29.0`, the version resolved by WTS. The SDK's `main` branch is v2 beta for a future protocol release, while v1 remains the supported production release ([current SDK README](https://github.com/modelcontextprotocol/typescript-sdk/blob/main/README.md), [`v1.29.0` source](https://github.com/modelcontextprotocol/typescript-sdk/tree/v1.29.0)).

## Conclusions

### 1. Optional authorization on one endpoint

**Yes at the HTTP/protocol boundary, but MCP does not define a named progressive-authorization mode.** Streamable HTTP requires one MCP endpoint supporting POST and GET. Separately, MCP authorization is optional, and the authorization error rules distinguish requests that require authorization from other requests. Nothing requires every operation at an MCP endpoint to have the same access policy. A server can therefore allow an anonymous request and require or evaluate a bearer token on another request to the same endpoint ([Streamable HTTP](https://modelcontextprotocol.io/specification/2025-11-25/basic/transports#streamable-http), [Authorization: protocol requirements](https://modelcontextprotocol.io/specification/2025-11-25/basic/authorization#protocol-requirements), [Authorization: error handling](https://modelcontextprotocol.io/specification/2025-11-25/basic/authorization#error-handling)).

The important qualification is that authorization is transport-level, not negotiated as an MCP capability. When a client uses a token, it **must include it on every HTTP request**, including requests in one logical session ([Access token usage](https://modelcontextprotocol.io/specification/2025-11-25/basic/authorization#access-token-usage)).

The TypeScript SDK can carry this design: `WebStandardStreamableHTTPServerTransport.handleRequest` accepts optional `authInfo` and passes it to handlers. Its supplied `requireBearerAuth` middleware is not optional-auth middleware: it rejects a missing header. Optional bearer handling therefore needs application middleware or request handling before the transport, as WTS already does for mandatory auth ([Web-standard transport source](https://github.com/modelcontextprotocol/typescript-sdk/blob/v1.29.0/src/server/webStandardStreamableHttp.ts), [bearer middleware source](https://github.com/modelcontextprotocol/typescript-sdk/blob/v1.29.0/src/server/auth/middleware/bearerAuth.ts)).

### 2. Discovery after anonymous initialization

**A successful anonymous `initialize` does not initiate or advertise MCP authorization.** The standardized flow begins when a protected request returns `401 Unauthorized` with a Bearer `WWW-Authenticate` challenge identifying Protected Resource Metadata. Clients then discover the authorization server and obtain a token. The specification also requires clients to support the protected-resource well-known locations, but it does not require proactive discovery after a successful anonymous initialization ([Protected Resource Metadata discovery](https://modelcontextprotocol.io/specification/2025-11-25/basic/authorization#protected-resource-metadata-discovery-requirements), [official authorization walkthrough](https://modelcontextprotocol.io/docs/tutorials/security/authorization#the-authorization-flow-step-by-step)).

`InitializeResult` has no authorization capability or authorization URL. Its capability negotiation covers protocol feature categories such as tools and resources ([Lifecycle: initialization and capabilities](https://modelcontextprotocol.io/specification/2025-11-25/basic/lifecycle#initialization)). The official TypeScript client reflects this: it sends an already-held token if its provider has one, starts OAuth after a `401`, and attempts scope step-up after a qualifying `403`; an ordinary successful response does not start authorization ([Streamable HTTP client source](https://github.com/modelcontextprotocol/typescript-sdk/blob/v1.29.0/src/client/streamableHttp.ts)).

Consequently, after anonymous initialization a client can become authorized only if it has an out-of-band/preconfigured token, deliberately starts a new authenticated connection, or later receives a proper `401`/`403` challenge from an operation it attempts. If private tools are omitted from the anonymous list, MCP provides no standard signal saying "authenticate to reveal more," so generic clients cannot be assumed to upgrade automatically.

### 3. Principal-specific lists, notifications, and sessions

**`tools/list` and `resources/list` may be principal-specific.** The protocol defines them as discovery of available tools/resources and does not require identical results for all clients. It also requires proper tool access controls and recommends access checks for sensitive resources, so filtering discovery does not replace authorization at `tools/call` or `resources/read` ([Tools: listing and security](https://modelcontextprotocol.io/specification/2025-11-25/server/tools#listing-tools), [Resources: listing and security](https://modelcontextprotocol.io/specification/2025-11-25/server/resources#listing-resources)).

List-change notifications have narrower semantics. A server declaring `tools.listChanged` or `resources.listChanged` says it can emit the corresponding notification; when that server's available list changes, it should emit `notifications/tools/list_changed` or `notifications/resources/list_changed` ([Tools: list changed](https://modelcontextprotocol.io/specification/2025-11-25/server/tools#list-changed-notification), [Resources: list changed](https://modelcontextprotocol.io/specification/2025-11-25/server/resources#list-changed-notification)). The high-level SDK advertises `listChanged: true` for registered tools/resources and can send these notifications, but its default list handlers enumerate the enabled registrations on that `McpServer`; they do not provide a built-in per-principal tool filter. Building a separately registered server for the request, or installing lower-level handlers, supplies that filtering ([`McpServer` source](https://github.com/modelcontextprotocol/typescript-sdk/blob/v1.29.0/src/server/mcp.ts)).

Two lifecycle constraints remain:

- Feature capabilities are negotiated during initialization, and both parties must use only negotiated capabilities. An anonymous initialization that advertises `tools` can later return a different tool list, but introducing `resources` later is not equivalent to a resource-list change if `resources` was never negotiated. That needs a newly initialized logical connection or an initialization that already advertises the category ([Lifecycle: capability negotiation](https://modelcontextprotocol.io/specification/2025-11-25/basic/lifecycle#capability-negotiation)).
- Stateful sessions must not be used as authentication. Every request must be re-authorized, and session IDs should be bound to user-specific information. Changing principal within one stateful session risks capability/list inconsistency and session confusion; if a list truly changes in that continuing session, the declared list-change notification applies ([Access token usage](https://modelcontextprotocol.io/specification/2025-11-25/basic/authorization#access-token-usage), [Security best practices: session hijacking](https://modelcontextprotocol.io/docs/tutorials/security/security_best_practices#session-hijacking)).

### 4. Stateless per-request server implications

SDK stateless mode (`sessionIdGenerator: undefined`) emits no session ID and performs no session validation. In v1.29.0 a stateless transport **must not be reused**: `handleRequest` throws after its first request. The official example creates a fresh `McpServer` and transport per POST and returns 405 for GET and DELETE ([transport source](https://github.com/modelcontextprotocol/typescript-sdk/blob/v1.29.0/src/server/webStandardStreamableHttp.ts), [official stateless example](https://github.com/modelcontextprotocol/typescript-sdk/blob/v1.29.0/src/examples/server/simpleStatelessStreamableHttp.ts)).

This supports request-scoped registration based on a validated token and avoids carrying one principal's in-memory server state into another request. It also means, absent separate shared state/routing, that subscriptions, resumability, standalone SSE delivery, and list-change notifications cannot persist across requests. A client must explicitly re-list, or reconnect and reinitialize when its authorization context changes. Rebuilding the server does not remove the client's lifecycle contract: capabilities and definitions still need to remain coherent with the initialization completed for that logical connection.

At the `2cce0be` pre-feature baseline, WTS's [`src/routes/api/mcp.ts`](../../src/routes/api/mcp.ts) followed the SDK's fresh-instance requirement: it authenticated first, then created a server and stateless web-standard transport for each request, with JSON responses enabled. Its behavior was mandatory `program:read` authorization, before the separate anonymous/public endpoint and precise private scopes were implemented.

### 5. Preconfigured long-lived bearer tokens

**They are compatible with core MCP/Streamable HTTP framing, but may be outside the standardized MCP Authorization profile.** The wire requirement is an `Authorization: Bearer <access-token>` header. The SDK client can send an existing provider token or custom request headers, and the server transport accepts application-supplied auth information; neither transport requires that discovery occur during that connection ([Access token usage](https://modelcontextprotocol.io/specification/2025-11-25/basic/authorization#access-token-usage), [client transport source](https://github.com/modelcontextprotocol/typescript-sdk/blob/v1.29.0/src/client/streamableHttp.ts), [`AuthInfo` source](https://github.com/modelcontextprotocol/typescript-sdk/blob/v1.29.0/src/server/auth/types.ts)). MCP itself does not state a maximum token lifetime.

However, conformance to MCP's HTTP authorization profile is more than using the Bearer syntax. A conforming protected server implements Protected Resource Metadata; clients must use tokens issued by the server's authorization server; and the server must validate that a token is intended for its own resource/audience ([Authorization overview](https://modelcontextprotocol.io/specification/2025-11-25/basic/authorization#overview), [Token handling](https://modelcontextprotocol.io/specification/2025-11-25/basic/authorization#token-handling)). A preconfigured OAuth access token can skip discovery in that client while still satisfying those rules. A custom API token with no MCP OAuth metadata/authorization-server relationship is instead a custom, out-of-band authentication convention: core MCP messages still work for preconfigured clients, but generic OAuth discovery/interoperability is not provided and it should not be described as implementing the MCP Authorization flow.

SDK detail: `AuthInfo.expiresAt` is optional, but the SDK's ready-made `requireBearerAuth` middleware rejects a token without an expiration. A non-expiring custom token therefore does not work with that helper unchanged ([`AuthInfo`](https://github.com/modelcontextprotocol/typescript-sdk/blob/v1.29.0/src/server/auth/types.ts), [bearer middleware](https://github.com/modelcontextprotocol/typescript-sdk/blob/v1.29.0/src/server/auth/middleware/bearerAuth.ts)).

### 6. Fail-closed behavior and caveats

The proposed distinction is protocol-compatible with these qualifications:

- No `Authorization` header may select the public view. A supplied malformed, expired, revoked, wrong-audience, or otherwise invalid token should return **401**, not silently fall back to public access. **403** is for a valid token with insufficient scope/permission; invalid `Origin` is a separate transport-level 403 case ([Authorization errors](https://modelcontextprotocol.io/specification/2025-11-25/basic/authorization#error-handling), [Streamable HTTP security warning](https://modelcontextprotocol.io/specification/2025-11-25/basic/transports#security-warning)).
- Standard OAuth-capable clients need a useful challenge. A 401 should identify `resource_metadata`; an insufficient-scope 403 should use `error="insufficient_scope"` and should identify the required scope and resource metadata. At the `2cce0be` baseline WTS returned only `Bearer realm="wts-mcp"`; the implemented custom bearer endpoint now adds insufficient-scope details but still does not publish standardized Protected Resource Metadata or OAuth discovery ([scope challenge handling](https://modelcontextprotocol.io/specification/2025-11-25/basic/authorization#scope-challenge-handling)).
- If an anonymous caller directly names a private tool or resource, the protocol does not decide whether to conceal it as unknown/not found or return a 401 challenge. Concealment avoids disclosing existence but provides no authorization-discovery trigger; a challenge enables the standard flow. Either path must still fail closed ([Tools: error handling](https://modelcontextprotocol.io/specification/2025-11-25/server/tools#error-handling), [Authorization errors](https://modelcontextprotocol.io/specification/2025-11-25/basic/authorization#error-handling)).
- List filtering is not the security boundary. Direct calls/reads must independently enforce the validated principal and scopes. Servers must validate tool inputs, resource URIs, and resource permissions ([Tools security](https://modelcontextprotocol.io/specification/2025-11-25/server/tools#security-considerations), [Resources security](https://modelcontextprotocol.io/specification/2025-11-25/server/resources#security-considerations)).
- Tokens must be validated for this MCP resource and must not be passed through from or to a downstream service. The official security guidance explicitly forbids token passthrough ([Token handling](https://modelcontextprotocol.io/specification/2025-11-25/basic/authorization#token-handling), [token-passthrough guidance](https://modelcontextprotocol.io/docs/tutorials/security/security_best_practices#token-passthrough)).
- Streamable HTTP requires validation of every supplied `Origin` header to prevent DNS rebinding. CORS response headers are not Origin validation. The `2cce0be` baseline route set permissive CORS headers without validating `Origin`; both implemented endpoints now use the same explicit Origin decision for request rejection and CORS response headers ([Streamable HTTP security warning](https://modelcontextprotocol.io/specification/2025-11-25/basic/transports#security-warning), [transport source](https://github.com/modelcontextprotocol/typescript-sdk/blob/v1.29.0/src/server/webStandardStreamableHttp.ts)).
- Authenticated list responses must not be reused as anonymous responses by application or intermediary caching, and bearer tokens require HTTPS and careful secret storage. This follows from the specification's token-theft and communication-security requirements ([Authorization security considerations](https://modelcontextprotocol.io/specification/2025-11-25/basic/authorization#security-considerations)).

## Overall finding

One stateless Streamable HTTP endpoint can serve anonymous public MCP features and a larger bearer-scoped feature set with the official TypeScript SDK. The principal interoperability limitation is discovery: anonymous initialization succeeds without any standard indication that authentication would reveal more, so expansion is dependable only for clients already configured with a token or clients that encounter a later standards-compliant authorization challenge. Custom bearer tokens preserve core MCP wire compatibility but do not, by themselves, implement MCP's OAuth authorization profile.
